# Min-Messenger

Min-Messenger - экспериментальный TCP-мессенджер на C++20. В текущем виде проект
содержит две разные серверные реализации:

- `main.cpp` - ранний сервер без протокола Secret: принимает TCP-клиентов и
  пересылает полученные байты всем остальным подключенным клиентам.
- `server.cpp`, `client.cpp`, `secret.cpp`, `secret.h` - текущая попытка перейти
  на собственный фреймовый протокол Secret поверх TCP.

Ниже описан протокол Secret по тому, как он сейчас реализован в коде, а не как
он должен выглядеть в идеальной версии.

## Secret

Secret - бинарный протокол поверх TCP. Он добавляет к каждому сообщению
фиксированный заголовок:

```text
+----------------+--------------+------------------+
| length: uint16  | code: uint8  | payload: bytes   |
+----------------+--------------+------------------+
| 2 байта         | 1 байт       | length - 3 байта |
+----------------+--------------+------------------+
```

### Транспорт

- Используется `AF_INET`/`SOCK_STREAM`, то есть IPv4 TCP.
- Сервер слушает `127.0.0.1:4551`.
- Клиент подключается к `127.0.0.1:4551`.
- Протокол не содержит собственного механизма установки соединения: сначала
  создается TCP-соединение, затем стороны обмениваются Secret-кадрами.

### Формат кадра

Каждый Secret-кадр состоит из:

1. `length` - 2 байта, тип `uint16_t`.
2. `code` - 1 байт, тип `uint8_t`.
3. `payload` - произвольные байты.

`length` хранит полный размер кадра в байтах, включая само поле `length`, поле
`code` и payload:

```text
length = 2 + 1 + payload_size
```

Например, сообщение с кодом `1` и payload `"ok"` имеет размер:

```text
length = 3 + 2 = 5
```

Логическая структура такого кадра:

```text
05 00 01 6f 6b
```

При этом порядок байтов для `length` сейчас не нормализован в сетевой порядок.
Код записывает `uint16_t` напрямую в память:

```cpp
*reinterpret_cast<uint16_t *>(dataforcend) = 3 + numbits;
```

Поэтому фактический порядок байтов зависит от архитектуры машины. На обычных
x86/x86_64 это little-endian. Между машинами с разным endian-форматом протокол
сейчас несовместим.

### Отправка кадра

Отправка реализована функцией:

```cpp
bool send_Secret(uint8_t code, const uint8_t *data, size_t numbits, int socket);
```

Параметры:

- `code` - код сообщения.
- `data` - указатель на payload.
- `numbits` - фактически количество байт payload, несмотря на название.
- `socket` - TCP-сокет.

Алгоритм:

1. Выделяется буфер размером `3 + numbits`.
2. В первые 2 байта записывается полный размер кадра.
3. В байт с индексом `2` записывается `code`.
4. Начиная с индекса `3` копируются байты payload.
5. Буфер отправляется через `send()` до тех пор, пока не будут отправлены все
   `3 + numbits` байт.

Если `send()` возвращает `<= 0`, счетчик неудачных попыток увеличивается.
После `MAX_SEND_ATTEMPTS` (`100`) функция возвращает `false`.

Обертка для текстовых сообщений:

```cpp
bool send_Message(uint8_t code, std::string message, int socket);
```

Она отправляет `message.c_str()` как payload. Завершающий `\0` в payload не
включается, потому что используется `message.size()`.

### Чтение кадра

Чтение реализовано функцией:

```cpp
bool read_Secret(
    uint8_t *data,
    size_t max_numbits,
    int socket,
    uint8_t &code,
    uint16_t &realmes
);
```

Параметры:

- `data` - пользовательский буфер для принятого содержимого после поля
  `length`.
- `max_numbits` - заявленный максимальный размер буфера.
- `socket` - TCP-сокет.
- `code` - выходной параметр, код сообщения.
- `realmes` - выходной параметр, полный размер кадра из поля `length`.

Фактический алгоритм:

1. `recv(socket, &msize, 2, 0)` читает первые 2 байта в `uint16_t msize`.
2. `realmes = msize`.
3. Дальше функция читает еще `msize - 2` байт.
4. Эти байты записываются в `data + total`, где `total` изначально равен `2`.
5. Когда накоплено `msize` байт, функция делает `code = data[0]` и возвращает
   `true`.

С этим связана важная особенность текущей реализации: после чтения `length`
оставшаяся часть кадра должна логически начинаться с `code`, но функция пишет
ее не в `data + 0`, а в `data + 2`. Затем код берется из `data[0]`, который
функция не заполняет. Из-за этого `read_Secret()` сейчас не извлекает `code`
надежно.

Если описывать ожидаемую структуру буфера после исправления, она должна быть
такой:

```text
data[0]                = code
data[1..payload_size]  = payload
```

Но текущий код фактически пишет принятые байты так:

```text
data[0]                = не заполняется read_Secret()
data[1]                = не заполняется read_Secret()
data[2]                = code
data[3..]              = payload
```

При этом `code` читается из `data[0]`, поэтому handshake и дальнейшая обработка
могут работать нестабильно или не работать вообще.

Также `max_numbits` сейчас не используется для проверки размера входящего
сообщения. Если peer отправит `length`, превышающий размер буфера, возможна
запись за пределы буфера.

## Коды сообщений

В коде используются следующие значения `code`:

| Код | Отправитель | Payload | Текущее значение |
| --- | ----------- | ------- | ---------------- |
| `0` | Сервер      | `"ok"`  | Первый кадр сервера после подключения клиента |
| `1` | Клиент      | `"ok"`  | Ответ клиента на серверный кадр `0` |
| `2` | Клиент      | Не определен | Зарезервирован в серверном цикле, обработка не реализована |

Другие коды в текущей реализации не описаны и не обрабатываются.

## Handshake

Текущий Secret-handshake реализован в `server.cpp` и `client.cpp`.

### Шаг 1. TCP-подключение

Сервер:

1. Создает TCP-сокет.
2. Привязывает его к `127.0.0.1:4551`.
3. Вызывает `listen()`.
4. В цикле принимает клиентов через `accept()`.
5. Для каждого клиента создает `User` и запускает отдельный detached-поток
   `ClientCom(user)`.

Клиент:

1. Создает TCP-сокет.
2. Подключается к `127.0.0.1:4551`.
3. Запускает detached-поток `comserver()`.

### Шаг 2. Сервер отправляет приветствие

Сразу после подключения сервер отправляет:

```text
code    = 0
payload = "ok"
```

То есть логический кадр:

```text
length = 5
code   = 0
data   = "ok"
```

Клиент ожидает, что первый полученный Secret-кадр будет иметь `code == 0`.
Если код отличается, клиент закрывает сокет и печатает:

```text
PROTO_ERRORSERVER MUST SEND 0
```

### Шаг 3. Клиент отвечает подтверждением

После получения серверного `code == 0` клиент отправляет:

```text
code    = 1
payload = "ok"
```

Сервер ожидает, что следующий кадр клиента будет иметь `code == 1`.
Если код отличается, сервер закрывает сокет и печатает:

```text
Everything is bad, code ne 1
```

### Шаг 4. Основной цикл

После handshake:

- сервер постоянно вызывает `read_Secret()` для данного клиента;
- клиент постоянно вызывает `read_Secret()` для сообщений от сервера;
- полноценная логика обмена сообщениями еще не реализована.

В сервере есть заготовка:

```cpp
if (code == 2) {

}
```

Но код `2` пока ничего не делает.

## Модель пользователя на сервере

В `server.cpp` определена структура:

```cpp
struct User {
    int id;
    int socket;
    std::vector<int> room;
};
```

Сервер назначает пользователю числовой `id`:

- если есть свободные идентификаторы в `free_ids`, берется последний из них;
- иначе используется следующий `id`.

Поле `room` существует, но сейчас не используется.

Глобальный список `users` объявлен, но новый пользователь в него сейчас не
добавляется. `users_mutex` тоже объявлен, но не используется.

## Ограничения текущей реализации Secret

### Нет сетевого порядка байтов

`length` записывается и читается как `uint16_t` в native-endian формате
процессора. Для переносимого протокола нужно использовать `htons()` при
отправке и `ntohs()` при чтении.

### Ошибка смещения при чтении

`read_Secret()` после чтения первых 2 байт пишет остаток кадра в `data + 2`,
но потом берет код из `data[0]`. Из-за этого извлечение `code` некорректно.

Минимально ожидаемая логика чтения:

```cpp
recv(socket, &msize, 2, 0);
recv(socket, data, msize - 2, 0);
code = data[0];
```

Либо, если хочется хранить в `data` весь кадр вместе с длиной, тогда нужно
заполнять `data[0]` и `data[1]` полем `length`, а `code` читать из `data[2]`.

### Нет проверки размера буфера

Параметр `max_numbits` передается в `read_Secret()`, но не используется.
Перед чтением нужно проверять:

```text
msize <= max_numbits
```

или, если `data` должен хранить только часть после `length`:

```text
msize - 2 <= max_numbits
```

### Не обрабатывается закрытие соединения при чтении тела

Если `recv()` во внутреннем цикле вернет `0`, это означает закрытие соединения.
Сейчас такой случай не завершает цикл, из-за чего возможен бесконечный цикл.

### Нет валидации минимального размера кадра

Корректный Secret-кадр не может быть меньше 3 байт:

```text
2 байта length + 1 байт code
```

Сейчас `read_Secret()` не проверяет `msize >= 3`.

### Payload не терминируется нулем

`send_Message()` отправляет строку без завершающего `\0`. Это нормально для
бинарного протокола, но принимающая сторона не должна считать payload
C-строкой, пока сама не добавит нулевой байт в отдельный буфер.

### Нет аутентификации и шифрования

Название Secret не означает, что данные шифруются. Сейчас payload передается
открытым текстом поверх обычного TCP.

## Текущий статус проекта

Протокол Secret сейчас находится в стадии заготовки:

- формат кадра и функции отправки уже есть;
- handshake сервер-клиент описан кодами `0` и `1`;
- основной обмен сообщениями после handshake не реализован;
- есть критические ошибки в чтении кадра и проверке размеров;
- сборочная конфигурация требует внимания: в `CMakeLists.txt` цели сейчас
  составлены так, что `server.cpp` подключается в обе цели, а `main.cpp` и
  `server.cpp` оба содержат `main()`.

Перед развитием протокола стоит сначала исправить `read_Secret()`, определить
сетевой порядок байтов для `length`, добавить проверки размера кадра и явно
задокументировать назначение кодов сообщений.
