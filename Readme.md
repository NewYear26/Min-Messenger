# Min Messenger

Min Messenger — учебный TCP-мессенджер на C++20 для POSIX-систем. В репозитории
есть две версии сервера:

- `main.cpp` — простой сервер, пересылающий полученные байты другим клиентам;
- `server.cpp` — сервер с бинарным фреймовым протоколом Secret;
- `client.cpp` — консольный клиент протокола Secret;
- `secret.cpp` и `secret.h` — кодирование, отправка и чтение Secret-кадров.

Проект использует POSIX sockets, pthreads и `<unistd.h>`, поэтому рассчитан на
Linux. На Windows его следует собирать в WSL или другой полноценной
POSIX-совместимой среде. MinGW без дополнительного слоя совместимости не
подходит.

## Сборка и запуск тестов

Для полного набора проверок нужны Linux, компилятор с поддержкой C++20 и
Python 3:

```sh
sh run_tests.sh
```

Скрипт собирает и запускает unit-тесты протокола, оба сервера, клиент и
интеграционные TCP-тесты. Если установлен CMake, дополнительно проверяется
конфигурация из `CMakeLists.txt`.

Серверы используют фиксированные адреса:

| Программа | Адрес |
| --- | --- |
| Legacy server (`main.cpp`) | `127.0.0.1:4551` |
| Secret server (`server.cpp`) | `127.0.0.1:4552` |
| Secret client (`client.cpp`) | подключается к `127.0.0.1:4552` |

## Протокол Secret

Secret — бинарный фреймовый протокол поверх TCP. TCP передаёт поток байтов и не
сохраняет границы сообщений, поэтому каждое сообщение содержит длину полного
кадра.

### Формат кадра

```text
+----------------+-------------+----------------------+
| length (2 байта)| code (1 байт)| payload (0..65532)  |
+----------------+-------------+----------------------+
```

- `length` — полный размер кадра: два байта длины, один байт кода и payload;
- `code` — тип сообщения;
- `payload` — произвольные бинарные данные;
- минимальный корректный `length` равен 3;
- максимальный payload равен `UINT16_MAX - 3`, то есть 65532 байтам.

Поле `length` кодируется как беззнаковое 16-битное число в сетевом порядке
байтов (big-endian). При отправке используется `htons()`, при чтении —
`ntohs()`.

Например, кадр с `code = 1` и payload `ok` имеет длину 5:

```text
00 05 01 6f 6b
|---|  |  |---|
length code payload
```

Payload не завершается нулевым байтом и может содержать `\0`. Его размер всегда
равен `length - 3`.

### Коды сообщений

Текущая реализация использует следующие коды:

| Code | Направление | Назначение |
| ---: | --- | --- |
| `0` | сервер → клиент | начало handshake, payload `ok` |
| `1` | клиент → сервер | подтверждение handshake, payload `ok` |
| `2` | в обе стороны | пользовательское сообщение |

## Handshake

После TCP-подключения Secret server отправляет:

```text
code = 0, payload = "ok"
```

Клиент отвечает:

```text
code = 1, payload = "ok"
```

Сервер проверяет код ответа. Если код отличается от `1`, соединение закрывается.
Текущая реализация не проверяет содержимое handshake payload.

После handshake клиент отправляет пользовательские сообщения с кодом `2`.
Сервер должен пересылать их другим аутентифицированным клиентам.

## API Secret

### `send_Secret`

```cpp
bool send_Secret(
    uint8_t code,
    const uint8_t* data,
    size_t numbits,
    int socket
);
```

Функция формирует кадр и отправляет его в сокет. Возвращает `false`, если:

- ненулевой payload передан через `nullptr`;
- payload не помещается в 16-битное поле длины;
- отправка завершилась ошибкой после повторных попыток.

Для пустого payload допустим вызов `send_Secret(code, nullptr, 0, socket)`.

### `read_Secret`

```cpp
bool read_Secret(
    uint8_t* data,
    size_t max_numbits,
    int socket,
    uint8_t& code,
    uint16_t& realmes
);
```

Функция читает один полный кадр:

- `data` получает только payload, без `length` и `code`;
- `max_numbits` задаёт доступную ёмкость `data`;
- `code` получает код кадра;
- `realmes` получает полный размер кадра, включая трёхбайтовый заголовок.

Чтение заголовка и payload выполняется в циклах, поэтому корректно обрабатывает
фрагментацию TCP. Пустой payload (`realmes == 3`) допустим.

### `send_Message`

```cpp
bool send_Message(uint8_t code, std::string message, int socket);
```

Это строковая обёртка над `send_Secret()`. Она передаёт ровно `message.size()`
байт и сохраняет встроенные нулевые байты.

## Legacy server

Legacy server не использует кадры. Каждый результат `recv()` он пересылает
остальным подключённым клиентам как набор бинарных байтов. Границы таких
сообщений протоколом не определены: несколько отправок могут объединиться в
один `recv()`, а одна отправка — разделиться на несколько чтений.

## Известные ограничения

Проект остаётся учебным и пока не предназначен для эксплуатации:

- Secret server не добавляет прошедших handshake клиентов в `users`, поэтому
  пересылка сообщений ещё не работает;
- доступ к глобальным спискам клиентов не синхронизирован;
- отключённые клиенты не удаляются из списков;
- legacy server передаёт потоку адрес локального объекта `User`;
- legacy server пишет нулевой байт за буфером при чтении ровно 1024 байт;
- одиночный `send()` в legacy server не гарантирует отправку всех байтов;
- клиент начинает читать stdin параллельно с handshake;
- обработчики соединений не имеют таймаутов и могут навсегда ждать неполный
  кадр;
- отправка в закрытый сокет может вызвать `SIGPIPE`;
- конфигурация `CMakeLists.txt` содержит несколько `main()` в одной цели и
  требует исправления.

Подробный актуальный список находится в `tests/ERRORS.md`.

## Безопасность

Название Secret не означает наличие шифрования. Протокол передаёт данные
открытым текстом и не обеспечивает аутентификацию, целостность или защиту от
повторного воспроизведения. Для реального мессенджера необходимы TLS,
аутентификация пользователей, ограничения ресурсов и контролируемое завершение
соединений.
